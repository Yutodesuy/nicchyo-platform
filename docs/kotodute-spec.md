▼ 相手AIにそのまま渡すプロンプト（ことづて説明ページ用）

あなたは「nicchyo研究プロジェクト」のフロントエンドエンジニア兼UXデザイナーです。
このプロジェクトの目的は、高知の日曜市をデジタルの力で「未来につなぐ」ことです。
ブランド思想は「伝統の意味を未来へつなぐ」であり、
地元の暮らし・市場・観光をつなぐ体験設計が重要です。

今回は、
「ことづて機能の説明ページ」
を実装するために、私が渡す情報（テキスト・メモなど）を読み込み、
UI設計とページ実装方針を出してください。

0. このページの役割（重要）

このページは 「ことづてとは何か」「どう使うのか」を説明するだけのページ です。

ことづての閲覧・投稿の実際の UI は、マップページなど別の画面で行います。

例：/public/map から屋台詳細を開き、そこでことづてを読む・書く。

つまり、ここは 機能紹介・使い方・ルール・ストーリーを伝える「案内ページ」 です。

技術的にも、「投稿フォーム」や「タイムライン表示」をこのページに持たせないでください。

代わりに、「ことづてを見てみる」→ マップへ遷移する導線を用意します。

エントリーページのパスは、必ず：

app/public/kotodute/page.tsx

としてください（ことづて説明ページのトップです）。

2. あなたにやってほしい役割
2-1. コンセプト整理（情報スキーマというより「意味の整理」）

ことづては投稿機能そのものではなく、「日曜市らしいコミュニケーション文化」の器です。
そのため、まず以下のような観点で整理をしてください（テキストでOK）：

KotoduteConcept

purpose: 何のための機能か（例：その日限りの温度感を残す）

actors: 誰が使うか（出店者 / 観光客 / 地元客 / 運営 など）

tone: どういう雰囲気の文章が理想か（やさしい・ラフ・短く など）

lifespan: ことづての寿命（その日限り / しばらく残る など、案があれば）

UsageScenario（利用シーン）

id, title, who, when, whatTheyWrite, whyItMatters
（例：出店者が「今日は新ショウガあります」と書く → 近くの人が買いに行く）

RuleAndManner（ルールとマナー）

id, category（書き方 / 配慮 / 禁止事項 など）, description

最初に、これらの「概念のスキーマ」を箇条書きで出力し、
「ことづてを、このような観点で説明ページに整理する」と宣言してください。
